# PIC32MX350F256H QEMU validation using Harmony CSP apps

This note ties **Microchip MPLAB Harmony 3** CSP sample apps to **QEMU `pic32mx3-generic`** (MIPS32 M4K, PIC32MX350-style model in-tree). Use it as a **checklist** for functional accuracy; many peripherals are **not** fully modeled yet.

## 1. Source layout (local clones)

Recommended clone root (example):

```bash
mkdir -p /data/projects/tmp/harmony_csp && cd /data/projects/tmp/harmony_csp
git clone https://github.com/Microchip-MPLAB-Harmony/csp.git
git clone https://github.com/Microchip-MPLAB-Harmony/csp_apps_pic32mx.git
```

Harmony apps expect the **CSP pack** (`csp`) on disk; MCC/manifest paths are usually relative to the **`.X`** project folder. Open the project in **MPLAB X** once so MCC can resolve `csp` (or set the Harmony content path in MCC preferences).

## 2. CLI build — important limitation

Upstream **`csp_apps_pic32mx`** commits typically include **`Makefile`** stubs that **include**:

- `nbproject/Makefile-impl.mk`
- `nbproject/Makefile-variables.mk`

Those files are **generated by MPLAB X / NetBeans** when the project is opened or built in the IDE. A **bare `git clone` + `make`** fails with:

```text
nbproject/Makefile-variables.mk: No such file or directory
```

**Practical workflow:**

1. Open the desired **`.X`** project in **MPLAB X v6** with **XC32** and **Harmony 3 / CSP** installed.
2. Let the IDE **generate** `nbproject/Makefile-impl.mk` and `nbproject/Makefile-variables.mk` (or use **Clean and Build**).
3. Then the documented command works from the firmware directory:

```bash
cd csp_apps_pic32mx/apps/<PERIPHERAL>/<APP>/firmware/<BOARD>.X
export PATH="/opt/microchip/xc32/v4.60/bin:$PATH"   # adjust version
make -f Makefile CONF=<configuration_name> build
```

The **`CONF=`** value must match a configuration name in `nbproject/configurations.xml` (often the same as the board suffix, e.g. `pic32mx470_curiosity`).

## 3. Name mapping (requested tests → upstream repo paths)

The exact folder names in the GitHub repo differ slightly from the names in some internal briefs:

| Your target | Closest app in `csp_apps_pic32mx` |
|-------------|-----------------------------------|
| UART echo on `-serial` | `apps/uart/uart_echo_blocking/` or `uart_echo_interrupt/` |
| DMA memory transfer | `apps/dmac/dmac_memory_transfer/` |
| ADC sampling | `apps/adc/adc_polled_mode/` or `adc_interrupt_mode/` (not `adcp_*` in this repo) |
| I²C EEPROM | `apps/i2c/master/i2c_eeprom/` |
| NVM read/write | `apps/nvm/flash_read_write/` |
| SPI EEPROM | `apps/spi/master/spi_eeprom_write_read/` |

Example **`.X`** roots (Curiosity board config):

- UART: `.../uart/uart_echo_blocking/firmware/pic32mx470_curiosity.X`
- DMAC: `.../dmac/dmac_memory_transfer/firmware/pic32mx470_curiosity.X`
- NVM: `.../nvm/flash_read_write/firmware/pic32mx470_curiosity.X`

## 4. PIC32MX350F256H vs Harmony board projects

Harmony sample **`.X`** projects in this repository are mostly configured for boards such as **PIC32MX470F512H** (see `mcc-config.mc4` `device:` field), not **PIC32MX350F256H**.

For **strict** QEMU `pic32mx3-generic` alignment:

- Regenerate or duplicate an MCC configuration for **PIC32MX350F256H**, **or**
- Accept **cross-part** binaries with caution (flash/RAM sizes, pinmux, **DEVID** reads, and missing peripherals may diverge).

QEMU’s **DEVID** register should match the **MX350** expectations used by `pic32mx3-generic`; a **470** image may branch differently if it checks device identity.

## 5. Expected QEMU behaviour vs current model (honest matrix)

| Area | Success criterion (idea) | QEMU `mips_pic32mx3.c` today |
|------|---------------------------|------------------------------|
| **SET/CLR/INV** | Per-word alias writes behave | Supported for many SFRs via `WRITEOP` pattern |
| **UART** | Echo on `-serial stdio` | UART1–U3 modeled; map the app to the host order: `-serial` #0 → guest **UART3**, #1 → **UART1**, #2 → **UART2** (`pic32mx3-generic`) |
| **Timer → ISR latency** | Deterministic enough for control | **Virtual clock** tied to host/TB execution; not cycle-accurate |
| **UART vs CPU timing** | Serial I/O does not “freeze” guest incorrectly | Char backend is async; if guest never runs ISR path, behaviour looks like a hang (firmware or IRQ model) |
| **DMAC** | Memory copy + IRQ | **Not modeled** as a full DMA engine in this PIC32 QEMU port — expect **failure** until implemented |
| **ADC** | Result registers update | **Partial / minimal** ADC plumbing; **stub channels** or extend model for tests |
| **I²C + EEPROM** | Handshake + data | **I2C** present; **external EEPROM** may need **I2C device model** or loopback stub |
| **NVM** | Flash persists after write | Currently **stub**: NVMCON **`WR`** clears; **no** persistent flash array for Harmony-style programming tests |

## 6. Suggested validation order (incremental)

1. **UART blocking echo** + **`-serial stdio`** after retargeting to **UART instance** QEMU exposes (or add a machine property for UART routing).
2. **Timer / core timer** apps (`tmr`, `coretimer`) if present — exercises IRQ + time without DMA.
3. **GPIO** polling/interrupt — checks basic IO and CN if used.
4. **NVM / DMA / ADC / I²C** — extend QEMU or narrow test to what is implemented.

## 7. Automation stub

After IDE-generated makefiles exist, you can loop builds from a script; until then, the blocker is **missing `nbproject/Makefile-*.mk`**, not XC32 itself.
